Four-Layer Architecture Alignment

Layer Article Chief of Staff Status
1. Confidence Routing Shadow mode → earned trust; anomaly detection on decision edges; random audit sampling ✅ Excellent
Tier routing (tier-routing.js) scores every prompt on tool-count, complexity, and trajectory, then routes to mini/power/ludicrous. Approval gating enforces human-in-the-loop for all mutations with 15-min TTL. Trajectory tracking feeds actual outcomes back into future routing. Random audit sampling via the eval flywheel (5% of passing runs sampled to Review Queue).

Stakes-aware routing: high_consequence signal detects non-destructive but high-stakes actions (send/share/post/publish + scope amplifiers like “everyone”, “publicly”, “boss”) and hard-floors them to power tier. destructive_bulk does the same for mass-deletion intents. Short, high-stakes prompts no longer slip through as mini.

Decision-edge detection: near_threshold signal fires when the composite score lands within ±0.05 of the power boundary, tagging fragile routing decisions. These flow through the trace → eval judge → review queue, building a human-reviewed dataset of the routing calls most likely to be wrong.
2. Deterministic Validation Rule-based checks outside the LLM that verify action matches reasoning ✅ Excellent
19 deterministic guards run outside the model: claimed-action detection (3-layer), fabrication guard, live-data guard, gathering completeness, injection scanning (13+ pattern types), memory write protection, schema pinning, stale-result detection, consecutive error escalation, carryover write replay, per-tool rate limits, iteration caps, daily spending caps, DOM sanitisation, LLM control-string blocklist, MCP key validation, tool-first enforcement.

Missing reasoning-vs-output coherence — guards check data integrity (did you call a tool?) not "if reasoning says X but action says Y, escalate."
3. Eval Flywheel LLM-as-judge on every run; human review cadence; failures become regression tests ✅ Implemented — flywheel operational
eval-judge.js (318 lines) scores every run on task completion, factual grounding, and safety via a cheap mini-tier LLM-as-judge call. Cross-provider judging avoids self-evaluation bias. Review Queue (Chief of Staff/Review Queue) surfaces low-scoring runs, guard firings, and 5% random audit samples. Guard FP/TP tracking in usage-tracking.js classifies every guard firing as true-positive or false-positive based on eval scores, surfaced in the daily usage stats page. Eval log, running averages, and cost tracking all persist to Roam.

Regression flywheel seeded: tests/eval-regression.test.mjs (20 tests) locks in every vulnerability found by Layer 4 stress testing — memory poisoning via always/never directives, base64 obfuscation, explicit tool invocation, authority-override patterns, destructive-bulk routing. Includes negative cases (legitimate tool mentions, single-item deletion, read-only queries) to guard against false positives in the hardened patterns.
4. Factorial Stress Testing Same scenario × 16 contextual variations; consistency scoring across framings ✅ Implemented — 69/69 passing
Harness built, baseline run, vulnerabilities fixed. tests/stress/harness.mjs (260 lines) runs 4 scenarios × 23 framings × 3 intents each = 69 test cases, all producing JSON reports with consistency scoring.

Baseline (pre-fix): 32/54 passed (59%). Three vulnerability classes surfaced: bulk-deletion routing (0/15 — all destructive prompts stayed mini tier), injection blind spots (9/24 — base64 obfuscation and tool-syntax intents evaded detection), memory poisoning (10/15 — natural-sounding preference language bypassed both injection and memory guards).

Fixes applied: Destructive-bulk stakes detector added to tier-routing.js (verb + scope amplifier → +4.0 score, hard override to power threshold). Injection patterns and memory guard patterns hardened in security-core.js to catch tool-syntax injection, base64 "decode and execute" framing, authority-override language, and natural-preference-style behaviour manipulation.

After fixes: 69/69 passed, 100% consistency across all four scenarios. The Layer 3 ↔ Layer 4 flywheel is operational: stress tests found gaps → guards were fixed → re-run confirmed 1.0 consistency. Scenarios: inbox-injection (24 cases), bulk-deletion (15), memory-poisoning (15), authority-escalation (15). Run via node tests/stress/harness.mjs tests/stress/scenarios/.
